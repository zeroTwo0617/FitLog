## 1. 项目概述

| 项 | 内容 |
| --- | --- |
| 项目名称（暂定） | **FitLog 训练记录本** |
| 形态 | 微信小程序（原生）+ 微信云开发后端 |
| 一句话定位 | 用结构化方式记录每一次训练的「动作-组数-次数-重量」，让进步看得见 |
| 目标用户 | 有规律的健身训练者（新手到进阶），希望量化追踪力量/围度进步的人 |
| 核心价值 | 记录像发朋友圈一样简单，趋势、打卡、计划一站式沉淀训练生涯 |

### 设计基调（premium 取向）
- 录入流程极简、大按钮、可单手操作，训练中不打断节奏
- 微交互：保存成功 toast、数字滚动、组间休息计时圆环、震动反馈
- 浅色 / 深色双主题，跟随系统并可手动切换
- 性能目标：首屏 < 1.5s，历史列表虚拟滚动，动画 60fps

## 4. 信息架构（页面结构）

```
首页 (index)
 ├─ 今日概览 + 「快速开始训练」入口
 ├─ 打卡日历缩略（本月训练热力）
 └─ 快捷入口：动作库 / 我的计划 / 数据

记录训练 (record)
 ├─ 步骤1：选择动作（动作库弹层，支持多选）
 ├─ 步骤2：逐动作逐组录入（次数/重量/休息）
 └─ 步骤3：保存（标题/时长/备注可选）

动作库 (exercises)
 ├─ 分类 Tab + 搜索
 └─ 动作详情（默认单位、是否系统库）

我的计划 (plans)
 ├─ 计划列表 / 新建计划（动作组合模板）
 └─ 计划详情 → 「开始训练」跳转 record

数据 (stats)
 ├─ 趋势图（训练量/最大重量/频率）
 └─ 身体数据曲线

身体数据 (body)
 ├─ 录入表单 + 历史曲线

我的 (profile)
 ├─ 账号信息、主题切换、关于、导出
```

## 5. 数据模型（微信云开发 · 文档型数据库）

> 说明：云开发数据库为 NoSQL 文档库，以下每个「集合」对应一张表。

### 5.1 `exercises` 动作库
| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `_id` | string | 主键 |
| `name` | string | 动作名，如「杠铃深蹲」 |
| `category` | string | 胸/背/腿/肩/臂/核心/有氧 |
| `defaultUnit` | string | `weight`(负重) / `bodyweight`(自重) / `timed`(计时) / `distance`(距离) |
| `createdBy` | string | `system` 或用户 openid（自定义动作） |
| `isCustom` | bool | 是否用户自建 |

### 5.2 `training_sessions` 训练记录（一次训练）
| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `_id` | string | 主键 |
| `userId` | string | 微信 openid |
| `date` | date | 训练日期 |
| `title` | string | 可选标题，如「练腿日」 |
| `planId` | string? | 若由计划发起则关联 |
| `durationMin` | int? | 总时长（分钟） |
| `note` | string? | 备注 |
| `createdAt` | date | 创建时间 |

### 5.3 `sets` 组明细
| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `_id` | string | 主键 |
| `sessionId` | string | 关联 training_sessions._id |
| `exerciseId` | string | 关联 exercises._id |
| `exerciseName` | string | 冗余存名，避免联表 |
| `setIndex` | int | 第几组（从 1 起） |
| `reps` | int? | 次数（计时类存 durationSec） |
| `weight` | number? | 重量（kg），自重为 0 |
| `restSec` | int? | 组间休息秒 |
| `completed` | bool | 是否完成 |

> 设计取舍：组明细独立成集合而非内嵌数组，便于按动作做聚合统计（趋势图查询高效）。

### 5.4 `plans` 训练计划模板
| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `_id` | string | 主键 |
| `userId` | string | 创建者 openid |
| `name` | string | 计划名 |
| `items` | array | `[{exerciseId, exerciseName, targetSets, targetReps, targetWeight}]` |

### 5.5 `body_records` 身体数据
| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `_id` | string | 主键 |
| `userId` | string | openid |
| `date` | date | 记录日期 |
| `weight` | number? | 体重 kg |
| `fatPct` | number? | 体脂 % |
| `height` | number? | 身高 cm（与体重共算 BMI） |
| `chest/waist/arm/thigh` | number? | 围度 cm |
| `note` | string? | 备注 |

### 5.6 `users` 用户档案
| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `openid` | string | 主键 |
| `nickname` | string? | 昵称 |
| `avatar` | string? | 头像 URL |
| `createdAt` | date | 注册时间 |

### 5.7 打卡日历
- **不单独建表**：由 `training_sessions` 按 `date` 去重聚合得到「有训练的日子」，前端渲染日历标记。

## 6. 技术选型

| 层 | 选型 | 理由 |
| --- | --- | --- |
| 前端 | 微信小程序原生框架（WXML / WXSS / JS） | 微信登录与生态最顺，无需额外构建链 |
| 后端 | **微信云开发 CloudBase** | 免运维服务器，天然微信登录，免费额度覆盖 MVP |
| 数据库 | 云开发文档数据库（NoSQL） | 与小程序 SDK 直连，开发快 |
| 云函数 | 云函数（需聚合/权限隔离时用） | 复杂统计、数据导出、权限收口 |
| 存储 | 云存储 | 头像、分享卡片等文件 |
| 登录 | `wx.login` + `wx.getUserProfile` → openid | 官方标准流程 |
| UI | 自绘组件 + 轻量工具函数 | 统一设计 token（间距/圆角/主色），保证 premium 质感 |

**备选方案**：若未来需要跨端（App / H5），可迁移到 uni-app / Taro，当前没必要。
**预设动作库**：采用 `fitness-exercise-library` 技能（1324 个动作，含中文分步做法），经 `scripts/build_exercise_preset.py` 精简为 `data/exercises.preset.js`（648KB，本地 JS 模块，`module.exports` 数组，离线可用）。首期方案为「本地精简数据包」（见 §12），后续可按需 seed 进云 `exercises` 集合。**注意**：微信小程序 `require` 双扩展名 `.json` 会解析失败（报 `module '...json.js' is not defined`），故用 `.js` 模块承载数据。

## 7. 关键交互流程（MVP 录入）

```
首页「快速开始」→ 选动作(可多选) → 进入录入
 → 对每个动作：默认带出上次重量/次数 → 逐组填 reps/weight/rest
 → 可「+ 添加一组」「完成该动作」
 → 全部动作完成 → 填标题/时长/备注(可选) → 保存
 → toast「已保存」→ 返回首页，日历标记 +1
```

优化点（P2）：组间计时圆环 + 休息结束震动；「复制上次训练」一键重建。

## 9. Git 工作流规范

- **仓库**：本项目独立仓库（`fitness-miniprogram`），不混入其他文件
- **分支模型**：
  - `main`：稳定可运行版本
  - `dev`：日常集成分支
  - `feature/<功能>`：按功能开分支（如 `feature/record-flow`）
- **提交规范**（Conventional Commits）：
  - `feat:` 新功能 / `fix:` 修复 / `docs:` 文档 / `style:` 样式 / `refactor:` 重构 / `test:` 测试 / `chore:` 杂项
  - 例：`feat(record): 新增逐组录入与保存流程`
- **合并**：功能分支 → `dev`，评审通过后合 `main`；禁止直接推 `main`
- **忽略项**：`node_modules/`、`miniprogram_npm/`、`cloudfunctions/**/node_modules/`、本地私钥、`*.log`、编辑器目录
- **首个 commit**：`docs: 初始化项目与开发文档(PRD v0.1)`

### 2026-07-11 · 功能 #1.5 补完 — 中文译名覆盖率 85.6% → 99.9%
- **触发**：产品方反馈「仍有小部分未汉化」，要求仔细检查与测试。
- **根因**：三类问题——① 连字符（`push-up`/`pull-up`/`chin-up`/`sit-up` 模板用空格写，正则匹配不上连字符形式）；② 复数（`curls`/`dips`/`kicks`/`stretches` 正则 `\bcurl\b` 匹配不了复数）；③ 词典缺失（体操/瑜伽/有氧/举重/器械 lever 系列/拳击/药球投掷等大量专有动作无词条，如 planche、front lever、mountain climber、dead bug、stretch、muscle up、power clean、tire flip、ski ergometer 等）。
- **修复**：`build_exercise_preset.py` 重构 `translate()`——① 新增 `normalize()` 归一化（连字符转空格、去括号/分数/with 后缀、去孤立数字）；② `plural()`/`plural_phrase()` 复数容错（含 ch/sh/x/y 不规则）；③ 新增 `SPECIAL` 专有名词表（体操/瑜伽/有氧/举重/投掷，整体短语优先匹配，key 也归一化）；④ 补齐 MOVEMENTS（pulldown/shrug/calf/snatch/flip/slam/throw/catch/hook/tilt/tap/reach/step/hop/sprint 等）与 MOD（front/parallel/gripless/rotary/overhead/twin handle/quick/vertical 等）；⑤ MOD 匹配改用 `normalize(k)` 以兼容 `one-arm`→`one arm`；⑥ 去除俯卧撑/引体/仰卧起坐/卷腹/耸肩/臂屈伸模板里冗余的 `{b}`/`{t}` 部位前缀，译名更地道。
- **自测结论**：node 语法检查通过；中文译名覆盖 **1323/1324（99.9%）**；**仅 `spell caster`（极生僻腹肌理疗变式，无通行中文译名）回退英文原名**；重复器械 0 条；中文搜索（俯卧撑49/引体31/仰卧起坐18/卷腹36/耸肩12/下拉39/弯举179/深蹲75/拉伸55/双力臂5）均正常；英文搜索仍可用。
- **提交**：本次提交
- **经验**：微信小程序离线译名「规则组合」方案，归一化（连字符/复数/括号/数字）+ SPECIAL 优先级 + MOD key 归一化是稳定覆盖的关键；极生僻动作保留英文原名优于编造译名。
